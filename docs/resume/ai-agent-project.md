# AI Agent 开发简历项目写法

## 项目标题

校园公开信息采集 Agent 与 RAG 问答系统

## 简历 bullets

- 独立设计并上线 Django/Celery 信息采集 Agent，接入 35 个公开信息源，自动完成抓取、抽取、OCR、去重、摘要分类、索引与发布，沉淀 1800+ 条公开内容。
- 构建基于 Meilisearch 的 RAG 问答助手，支持流式回答、引用溯源、会话历史、token/费用统计和无预算 fallback，降低幻觉与不可控调用成本。
- 实现 Agent runtime 可观测链路，用 AgentRun/AgentStep 记录工作流状态、工具名、耗时、错误、LLM 成本和评测结果，便于定位抓取、检索和生成问题。
- 设计低成本自愈与 MCP 工具接口，支持站点健康检查、公开内容检索、自愈 dry-run、失败源重试和 RAG 索引修复，明确区分公开能力与后台管理边界。

## 面试讲解思路

1. 先讲业务问题：学校通知分散在官网、学院站、就业网和公众号链接里，人工筛选成本高、错过信息风险高。
2. 再讲 Agent 工作流：调度抓取 -> 列表发现 -> 正文抽取/OCR -> 去重 -> AI 摘要分类 -> 重要度排序 -> RAG chunk 索引。
3. 展示 `/agent/`：说明数据规模、运行记录、评测结果、成本控制和自愈机制。
4. 展示 `/ask/`：用示例问题验证检索、引用、流式回答、token/费用统计和 fallback。
5. 最后讲工程约束：2C2G 服务器、Docker Compose、MySQL/Redis/Meilisearch、预算熔断、后台权限和公开页面脱敏。

## 为什么这样写

- 大厂 AI Agent 实习更看重工程闭环：数据接入、RAG、工具调用、评测、可观测、上线和成本控制，比单纯调用大模型 API 更有区分度。
- 项目有真实用户入口和真实数据规模，能证明你做过线上系统，而不是一次性 demo。
- 简历 bullet 用数字和机制表达结果，面试时可以直接打开 `/agent/` 和 `/ask/` 逐项验证。

## 可以主动承认的限制

- 当前评测是轻量检索评测，主要覆盖命中率、关键词和引用覆盖；还没有构建人工标注的端到端答案质量集。
- 当前 LangGraph 作为组合点接入，复杂规划和多智能体协作还不是核心能力；项目重点是稳定的数据/RAG/工具闭环。
- 公众号和社媒账号主页没有做自动监控，第一版只处理具体公开链接，避免抓取合规和反爬风险扩大。
